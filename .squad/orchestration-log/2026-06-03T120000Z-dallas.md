## Objective

## Scope Definition (Per Lambert QA Review)

### Broken Flows (Required Fix)
1. **Custom API create** — Validation read from `useCustomApis().customapis` (solution-filtered), missed duplicates outside selected solution
2. **Catalog create-root** — Validation read from `useCatalogs().catalogs` (solution-filtered), missed duplicates outside selected solution
3. **Catalog create-category** — Same broken path as create-root

### Previously Sound (Regression Coverage)
- **Request Parameter create** — Uses `useCustomApiRequestParameters()` (scoped by CustomApiId, not solution)
- **Response Property create** — Uses `useCustomApiResponseProperties()` (scoped by CustomApiId, not solution)

### Missing Validation (New Implementation)
- **Catalog Assignment create** — No duplicate validation existed; required guard on (catalog + object-id + object-type) uniqueness

---

## Design Decision

**Validation Scope Ownership Model:**
- Custom API uniquename → all Custom APIs (global scope)
- Catalog uniquename → all Catalogs (global scope)
- Request Parameter uniquename → parameters under selected Custom API (parent scope)
- Response Property uniquename → response properties under selected Custom API (parent scope)
- Catalog Assignment duplicate → (selected catalog + object id + object type) uniqueness (parent scope)

**Browsing vs. Validation Principle:**
- Use solution-filtered collections for UI browsing/picker display (UX aid)
- Use authoritative full collections for uniqueness validation (data integrity boundary)
- Solution filtering is not a data boundary; it's a browsing aid
